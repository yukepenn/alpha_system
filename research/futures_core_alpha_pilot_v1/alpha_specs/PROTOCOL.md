# AlphaSpec Batch Protocol

Campaign: `ALPHA_FUTURES_CORE_ALPHA_PILOT_V1`  
Phase: `FUTCORE-P05`  
Protocol type: value-free AlphaSpec batch drafting contract  
Recorded on: 2026-06-07

This protocol is the canonical drafting contract for the `FUTCORE-P07` through
`FUTCORE-P11` AlphaSpec batches. It defines schema usage, field rigor, family
quotas, horizon/session/cost declarations, and critique independence rules. It
does not draft, approve, implement, run, promote, or reject any family
AlphaSpec.

## Source Contracts

The protocol binds to these existing contracts and does not redefine them:

- Governance schema:
  `src/alpha_system/governance/alpha_spec.py`
  (`ALPHA_SPEC_REQUIRED_FIELDS`, `ALPHA_SPEC_FIELD_TYPES`, and
  `ALPHA_SPEC_SUBSTANTIVE_FIELDS`).
- Campaign goal and phase plan:
  `campaigns/ALPHA_FUTURES_CORE_ALPHA_PILOT_V1/GOAL.md` and
  `campaigns/ALPHA_FUTURES_CORE_ALPHA_PILOT_V1/PHASE_PLAN.md`.
- Campaign policy blocks in
  `campaigns/ALPHA_FUTURES_CORE_ALPHA_PILOT_V1/campaign.yaml`, especially
  `agent_policy`, `horizon_session_policy`, and `research_budget_policy`.
- Cost contract:
  `research/futures_core_alpha_pilot_v1/cost_model/cost_model_contract.md`.
- Research interpretation policy:
  `docs/RESEARCH_INTERPRETATION_POLICY.md`.

Every draft remains value-free. Drafts may record ids, references, assumptions,
diagnostic requirements, limitations, and review criteria. They must not record
raw market data, provider responses, feature values, label values, per-row
diagnostics, costs, returns, alpha claims, profitability claims, tradability
claims, paper/live/broker readiness, production readiness, or capital
allocation language.

## Canonical AlphaSpec Payload

Each draft must contain exactly one governance `AlphaSpec` payload whose
top-level fields equal `ALPHA_SPEC_REQUIRED_FIELDS`. The payload must not add,
rename, omit, or fork top-level fields. Human notes may surround the payload in
the draft file, but the canonical payload is the only object that can be
validated as an `AlphaSpec`.

Template shape, with placeholders only:

```json
{
  "alpha_spec_id": "<aspec_24_hex_generated_from_payload>",
  "hypothesis_id": "<hyp_24_hex>",
  "target_instruments": ["<ES|NQ|RTY>"],
  "data_assumptions": {},
  "factor_inputs": ["<registry_or_existing_primitive_reference>"],
  "label_references": ["<lspec_24_hex>"],
  "exclusion_rules": ["<substantive exclusion rule>"],
  "timestamp_assumptions": {},
  "cost_assumptions": {},
  "expected_failure_modes": ["<substantive failure mode>"],
  "promotion_criteria": ["<substantive criterion for later review>"],
  "created_by": "Hypothesis Scout:<assignment_id>",
  "created_at": "<timezone-aware ISO-8601 timestamp>"
}
```

The placeholder template above is not an AlphaSpec draft and must not be copied
as a filled candidate without replacing every placeholder with reviewed,
schema-valid content.

## Required Field Contract

| Field | Schema type | Drafting rule |
| --- | --- | --- |
| `alpha_spec_id` | `str` | Must be an `aspec_<24-hex>` governance id generated by `generate_alpha_spec_id` from the complete payload components. Do not hand-craft semantic ids. |
| `hypothesis_id` | `str` | Must be a valid `hyp_<24-hex>` governance id for the originating hypothesis card or hypothesis record. |
| `target_instruments` | `list` | Must be a non-empty list containing only `ES`, `NQ`, and/or `RTY`. Cross-market drafts must include at least two of these instruments; other families may target one or more in-scope instruments. |
| `data_assumptions` | `dict` | Must be non-empty and substantive. Cite accepted DatasetVersion, FeaturePack, LabelPack, registry, session, coverage, and missingness assumptions by id or policy reference only; do not copy values or arbitrary raw paths. |
| `factor_inputs` | `list` | Must be a non-empty list of existing registry-resolved feature references, accepted feature-pack references, or existing primitive names. Missing inputs may be flagged as a later FeatureRequest need, but no feature is implemented or assumed available in the draft. |
| `label_references` | `list` | Must be a non-empty list of valid `lspec_<24-hex>` LabelSpec ids. Do not use placeholder label ids or final-session information as an intraday label input. |
| `exclusion_rules` | `list` | Must be non-empty and substantive. Include data-quality, timestamp, missingness, session, cost, and duplicate-exposure exclusions relevant to the family. |
| `timestamp_assumptions` | `dict` | Must be non-empty and substantive. Declare `available_ts` usage for every feature/input and `label_available_ts` usage for every label reference. Declare decision timestamp, bar/session timestamp semantics, exchange trade date handling, and session segment handling. |
| `cost_assumptions` | `dict` | Must be non-empty and substantive. Reference `CostModelVersion` `cmv_futcore_pilot_three_layer_session_stress_v1` and exactly the `zero_cost`, `base`, `stress_1`, `stress_2`, and `double_cost` profiles. |
| `expected_failure_modes` | `list` | Must be non-empty and substantive. Include family-specific failure modes plus no-lookahead, missingness, cost fragility, session instability, regime instability, overfit, and duplicate exposure where applicable. |
| `promotion_criteria` | `list` | Must be non-empty and substantive. These are criteria for later critic/reviewer assessment only; they do not approve or promote the draft. `zero_cost` evidence cannot satisfy any criterion. |
| `created_by` | `str` | Must identify the drafter role as `Hypothesis Scout` plus an assignment or task id. It must not name the critic, reviewer, promoter, diagnostics runner, or human capital decision role as drafter. |
| `created_at` | `str` | Must be a timezone-aware ISO-8601 timestamp recorded when the draft is created. Prefer UTC with a `Z` suffix. |

The schema's substantive fields are:

- `data_assumptions`
- `exclusion_rules`
- `timestamp_assumptions`
- `cost_assumptions`
- `expected_failure_modes`
- `promotion_criteria`

Substantive fields must not be empty, `TBD`, `unknown`, `placeholder`, terse,
or generic. They must be specific enough for an independent critic to decide
whether the draft is testable, bounded, and consistent with the campaign.

## ID And File Naming Convention

The governance ids inside the payload are canonical:

- `hypothesis_id` uses the governance `HypothesisCard` prefix `hyp`.
- `alpha_spec_id` uses the governance `AlphaSpec` prefix `aspec`.
- `label_references` use the governance `LabelSpec` prefix `lspec`.

The `alpha_spec_id` is content-derived. Generate it only after all component
fields except `alpha_spec_id` are final for the draft. Any substantive payload
change requires regenerating the `alpha_spec_id`.

Draft files in `FUTCORE-P07` through `FUTCORE-P11` should use deterministic
batch filenames for audit, for example:

```text
research/futures_core_alpha_pilot_v1/alpha_specs/<family>/<phase_id>_<family>_<nn>.md
```

The filename sequence is an audit index only. It is not a governance id and must
not be used as `alpha_spec_id`.

## Horizon And Session Rules

Every draft must declare its horizon and session assumptions in the substantive
fields, especially `data_assumptions`, `timestamp_assumptions`,
`cost_assumptions`, `expected_failure_modes`, and `promotion_criteria`.

Required horizon rules:

- Primary pilot horizons are `5m`, `10m`, `15m`, and `30m`.
- `1m` is sampling-only unless explicitly marked as execution-fragile
  diagnostics.
- `1m` and `3m` diagnostic views cannot be a promotion basis and require
  stricter spread, liquidity, cost, and thin-session limitations.
- Extended intraday horizons (`60m`, `120m`, `240m`, or `session_close`) may be
  referenced only as extended diagnostics with stronger sample, overlap, regime,
  and stability caveats.
- No modeled holding period may cross the exchange daily maintenance /
  trade-date break.

Required session-view list:

- `full_session`
- `RTH_only`
- `ETH_only`
- `ETH_evening`
- `ETH_overnight`
- `pre_RTH`
- `RTH`
- `post_RTH`
- `RTH_with_ETH_context`

Drafts must state which session views are in scope, which are diagnostic-only,
which are excluded, and why. ETH, pre-RTH, and post-RTH are research-in-scope
but not trading-approved and require thin-session cost and liquidity caveats.

## Timestamp And No-Lookahead Rules

Every draft must declare how the following are used:

- `available_ts` for each feature, factor input, session context field, and BBO
  input.
- `label_available_ts` for each label reference.
- Decision timestamp and whether the signal is formed at bar open, during bar,
  or after a completed bar.
- `exchange_trade_date` and session segment boundaries.
- Cross-market timestamp alignment when more than one instrument is involved.

Forbidden timestamp usage:

- No final-session high, low, VWAP, range, volume, or aggregate may be used
  intraday before the session is complete.
- No final RTH value may be used in an RTH intraday decision before RTH close.
- No final ETH value may be used before the ETH window is complete.
- No label information, future bar value, or same-bar outcome may be used as a
  feature.
- No arbitrary wall-clock joins that ignore `available_ts` or
  `label_available_ts`.

Allowed intraday session context must be named as running or completed context,
for example `running_vwap_so_far` or completed ETH context used by
`RTH_with_ETH_context`, and must remain point-in-time under `available_ts`.

## Cost Rules

Each draft's `cost_assumptions` must reference:

- `CostModelVersion`:
  `cmv_futcore_pilot_three_layer_session_stress_v1`
- Layer 1: hard transaction cost.
- Layer 2: spread crossing from valid point-in-time BBO where available.
- Layer 3: slippage and capacity proxy.
- Profiles: exactly `zero_cost`, `base`, `stress_1`, `stress_2`, and
  `double_cost`.
- Thin-session overlays for `ETH_only`, `ETH_evening`, `ETH_overnight`,
  `pre_RTH`, and `post_RTH`.

`zero_cost` is diagnostic-only. It must not appear as a favorable review
condition, a promotion basis, survivor evidence, or a reason to keep a draft
that fails required non-zero cost profiles. BBO, slippage, and capacity outputs
are proxy sensitivities only, not proof of tradability or fill capacity.

## Family-Specific Diagnostic Declarations

Each draft must declare the diagnostics the later StudySpec and runtime phases
would need. This phase records declarations only; no diagnostic is run here.

| Family | Mandatory diagnostic declarations |
| --- | --- |
| Cross-market / relative value | Cross-instrument `available_ts` alignment, bar/session alignment, lead/lag convention, missingness by instrument, asynchronous or stale input handling, beta/residual or spread-construction assumptions, and duplicate exposure to other cross-market ideas. |
| VWAP / session auction | Running-vs-final VWAP distinction, opening-range availability, overnight high/low availability, gap construction timing, session transition handling, and final aggregate lookahead rejection. |
| Regime-gated momentum vs reversion | Computable regime-activation logic, point-in-time regime inputs, whether the regime gates momentum or reversion, per-regime sample/coverage caveats, regime transition instability, and duplicate exposure to broad trend/volatility filters. |
| Liquidity sweep / failed breakout / objective PA | Objective computable rules for sweep, close-back-inside, wick rejection, displacement, compression, breakout, and failed-breakout conditions; no visual-only or subjective annotations; declare timestamped level availability and missing/ambiguous level exclusions. |
| BBO tradability / top-book confirmation | Valid BBO requirements, spread/microprice/imbalance/quote-quality framing, stale/crossed/missing quote exclusions, confirmation or risk-control role, and whether the idea is only a tradability/confirmation overlay rather than a standalone edge hypothesis. |

Volume/activity overlays must identify the host family, existing primitive or
pack reference, timestamp availability, and why the overlay is required. They
must not create a standalone family or standalone quota.

## Critique And Independence Rules

Every draft must be written so `FUTCORE-P12` can apply an independent critic
review without role ambiguity:

- `created_by` identifies only the `Hypothesis Scout` drafter.
- The AlphaSpec Critic must not draft or co-draft the spec it reviews.
- The semantic reviewer must be independent of both the drafter and critic.
- Any later promoter or promotion decision writer must be independent of the
  drafter, critic, reviewer, diagnostics runner, and implementer.
- The Diagnostics Runner cannot approve or promote the draft.
- Feature Engineer and Label Engineer cannot review their own implementation.
- Statistical Reviewer must be independent of the implementer.
- The human owns all capital, live, paper, broker, and production judgment.
- Self-review and self-promotion are forbidden.

Each batch must carry duplicate-exposure and rejected-idea awareness for later
critique. At minimum, drafts must state whether the hypothesis appears to reuse
the same exposure, label, session, horizon, or trigger family as another draft
or known rejected idea. Duplicate awareness is not automatic rejection, but
unstated duplicate exposure is a protocol failure.

## Critic Checklist For FUTCORE-P12

The AlphaSpec Critic should be able to answer all of the following from the
draft without reading code or running diagnostics:

- Does the payload validate against the governance `AlphaSpec` schema?
- Are all substantive fields specific and non-vague?
- Is the target universe limited to ES/NQ/RTY?
- Does the draft stay within its family quota and phase directory?
- Does it declare primary, fragile, and excluded horizons?
- Does it declare session views and maintenance-break flatness?
- Does it use `available_ts` and `label_available_ts` correctly?
- Does it forbid final-session aggregates in intraday decisions?
- Does it cite the P04 `CostModelVersion` and all required profiles?
- Is `zero_cost` diagnostic-only and excluded from promotion criteria?
- Does it declare mandatory family diagnostics?
- Does it identify expected failure modes and duplicate/rejected-idea exposure?
- Are drafter, critic, reviewer, and promoter roles separated?
- Does it avoid alpha, profitability, tradability, live, paper, broker,
  deployment, production, or capital-allocation claims?

Any missing answer is a revision request or rejection candidate for
`FUTCORE-P12`; this protocol does not itself approve or reject the draft.

# P094500_DAMAGED_PACK_RESTORE_PLAN: provenance-complete restore plan for clobbered packs

## Purpose

INCIDENT REPAIR (2026-06-12): three pack families have registry-vs-disk fver
mismatches after trimmed-scope force-recomputes (coordinator incident,
forensics in `~/logs` + session transcript): bbo_tradability_top_book
(10 stale ids × 24 partitions), session_calendar_maintenance (5 × 24:
session_calendar_roll_{bars_to_roll,day_of_week,halt_status_flag,
minutes_to_expiration,minutes_to_roll}), regime_volatility_compression
(2 × 24: pack-local base_ohlcv_{returns,rolling_range}). KEY COMPLICATION:
packs contain the UNION of multiple runs' outputs — the committed
session config (FUTSUB-P07) governs only 8 names; the roll countdowns came
from another phase's run (likely FUTSUB-P09 roll/maintenance guards); the
regime pack's stale ids are input copies outside its governed_scope. A naive
full-config re-run would AGAIN rewrite away the out-of-scope siblings. ALL
10 kill-shot sspecs lock stale fvers — the kill-shot is blocked until packs
are restored with REGISTRY-HASH-IDENTICAL content (code unchanged for these
features ⇒ deterministic recompute should reproduce content hashes;
hash-identity = locks stay valid = no re-lock cascade for non-bbo specs).

## Scope (in-bounds)

1. **Provenance reconstruction** (read-only over registry + repo): for EVERY
   stale fver (expect 408), resolve its `feature_request_id` /
   `materialization_plan_id` to the producing phase/config/command — use
   registry provenance tables (inspect schema), FUTSUB handoffs
   (handoffs/ALPHA_FUTURES_RESEARCH_SUBSTRATE_SCALEOUT_V1/FUTSUB-P07/P09/
   P10/P12/P13.md), and committed configs. Output: a value-free provenance
   table (fver → feature_id, partition, producing phase, config path or
   reconstruction, command template).
2. **Union restore strategy per pack family**: determine how each pack's
   FULL registered fver set can be regenerated in one guarded write per
   partition — either (a) an ordered sequence of the original runs where
   each later run PRESERVES prior content (verify the driver/materializer
   merge behavior — does any append/merge mode exist?), or (b) a
   coordinator-authored UNION config per family (all features the registry
   expects, with the inputs each needs — e.g. roll calendar refs for
   countdown features), validated against the family code's capabilities.
   The strategy MUST be compatible with the overwrite guard being added in
   P090500 (superset writes only). If the regime pack's base_ohlcv copies
   cannot be regenerated by the regime family code, identify exactly what
   wrote them (P10? the driver's input-staging?) and how to reproduce.
3. **Single-partition proof**: execute the restore for ONE partition per
   family (ES_2019) in a bounded run, then verify: every REGISTERED fver
   for that path present on disk AND `value_content_hash` MATCHES the
   registry for the previously-stale fvers (hash-identity proof —
   deterministic restoration). Report per-fver hash match/mismatch counts
   (value-free). A mismatch is a FINDING to report (code drift since
   original materialization), not something to paper over.
4. **Coordinator runbook**: `research/futures_substrate_scaleout_v1/repair/`
   PACK_RESTORE_RUNBOOK.md (value-free): exact full-grid commands per family
   (24 partitions each), expected durations (from the single-partition
   timing), verification command (the P090500 pack-audit tool if merged,
   else an equivalent check script committed under tools/futures_substrate_scaleout/),
   and the bbo note (bbo restore waits for the P085901 grid fix; its
   re-registration will carry NEW content by design → sspec_6088f0 re-lock
   required; the other families must verify hash-identity).
5. Do NOT run the full 24-partition grids (coordinator executes the runbook
   as background compute); do NOT touch feature family code, diagnostics,
   or locks.

## Hard constraints

- Read-only against the live registry except the single-partition proof
  runs (which write only the target pack files + registry rows exactly as
  the sanctioned CLI does — no manual SQLite edits ever).
- Full committed/union configs only — never a scope-subset write (the
  incident class). If the P090500 guard is already merged into this
  worktree's base, the proof runs must pass it.
- Explicit staging; no values/SQLite/runs committed; research-only language.
